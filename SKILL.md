# Picasso

The ultimate design skill for producing distinctive, production-grade frontend interfaces, visual artifacts, and design systems. Consolidates best practices from Anthropic's frontend-design, canvas-design, algorithmic-art, theme-factory, and web-artifacts-builder skills, Impeccable's 7-domain reference system and 20 steering commands, VoltAgent's 9-section DESIGN.md format (55 real-world sites), Vercel's 45 React performance rules, frontend-slides' 12 style presets, and specialized libraries for sound (soundcn), haptics (web-haptics), icons (better-icons), text animation (torph), avatar generation (facehash), and pixel typography (Geist Pixel).

Every output should look like it was built by a senior design engineer who spent days refining it, not generated by an AI in seconds.

---

## Configurable Settings

Three dials (1-10) control the overall character of the output. Adjust based on what is being built. The user can set these explicitly or they can be inferred from context.

- **DESIGN_VARIANCE** (default: 6) -- How experimental the layout is. 1-3: clean, centered, conventional. 4-6: considered asymmetry, intentional breaks. 7-10: avant-garde, overlapping elements, unconventional grids.
- **MOTION_INTENSITY** (default: 5) -- How much animation is present. 1-3: hover states and fades only. 4-6: staggered reveals, scroll-triggered, text morphing. 7-10: magnetic cursors, parallax, complex choreography.
- **VISUAL_DENSITY** (default: 5) -- How much content fits on one screen. 1-3: spacious, luxury, breathing room. 4-6: balanced, structured whitespace. 7-10: dense dashboards, data-heavy, compact.

When the user says "make it premium" or "luxury feel," drop VISUAL_DENSITY to 2-3 and MOTION_INTENSITY to 4-5. When they say "dashboard" or "admin panel," raise VISUAL_DENSITY to 7-8. When they say "make it pop" or "wow factor," raise DESIGN_VARIANCE and MOTION_INTENSITY to 7-8.

## Step 1: Design Thinking

Before writing a single line of code, answer these questions internally:

### Context Gathering
Confirm these before proceeding. Context must come from loaded instructions, the project's design system, or the user. Do not infer from code alone:
- **Target audience**: Who uses this and in what context?
- **Use cases**: What jobs are they trying to get done?
- **Brand personality/tone**: How should the interface feel?

### Design Direction
**Purpose.** What problem does this interface solve? Who uses it? What is the single most important action?

**Tone.** Commit to a specific aesthetic direction. Not "clean and modern" (that is meaningless). Pick something with teeth: brutalist, editorial, retro-terminal, luxury serif, toy-like, industrial, organic, Swiss grid, art deco, vaporwave, newspaper broadsheet, scientific journal, soft pastel, neon cyber, or something entirely original. The direction should be informed by the content, not applied generically.

**Differentiation.** What makes this unforgettable? What is the one thing someone will remember after closing the tab? If there is no answer, the design is not ready.

**Constraints.** Framework requirements, accessibility targets, performance budgets, existing design tokens.

## Step 2: Aesthetic Execution

### Typography
Choose fonts that are distinctive, not default. Never use Inter, Roboto, Arial, or system fonts as primary choices. Better alternatives: Instead of Inter use Instrument Sans, Plus Jakarta Sans, or Outfit. Instead of Roboto use Onest, Figtree, or Urbanist. Instead of Open Sans use Source Sans 3, Nunito Sans, or DM Sans. For editorial/premium: Fraunces, Newsreader, Lora.

Pair a display font with a body font. Use a modular type scale (1.25 or 1.333 ratio). Set line heights between 1.4 and 1.6 for body text. Use OpenType features (ligatures, tabular numbers, small caps) when the font supports them. Vertical rhythm: line-height is the base unit for ALL vertical spacing. Set `max-width: 65ch` on text blocks. See `references/typography.md`.

### Color
Build palettes with intention. Use OKLCH for perceptually uniform color manipulation. Always tint neutrals (never use pure gray, pure black, or pure white). Dominant color with sharp accent outperforms evenly distributed palettes. Test contrast ratios: 4.5:1 minimum for body text, 3:1 for large text and UI elements. The 60-30-10 rule: 60% neutral backgrounds, 30% secondary, 10% accent. Alpha/opacity is a design smell -- use actual color values instead. See `references/color-and-contrast.md`.

### Spatial Composition
Use a spacing scale (4px base: 4, 8, 12, 16, 24, 32, 48, 64, 96). Asymmetric layouts, overlapping elements, diagonal flow, and grid-breaking moments create visual interest. Generous negative space communicates confidence. Cards are not always required -- sometimes flat sections, dividers, or whitespace work better. Never nest cards inside cards. The Squint Test: blur your eyes -- can you identify the most important element? See `references/spatial-design.md`.

### Backgrounds and Visual Depth
Never default to flat solid colors. Create atmosphere with gradient meshes, noise textures, geometric patterns, layered transparencies, dramatic shadows, grain overlays, or subtle backdrop filters. Depth comes from elevation systems: consistent shadow scales, surface hierarchy, and z-index discipline. In dark mode, elevation goes lighter (higher surfaces are lighter), not darker.

### Motion
One well-orchestrated page load with staggered reveals creates more impact than scattered micro-interactions. Duration rule: 100-150ms for instant feedback, 200-300ms for state changes, 300-500ms for layout changes, 500-800ms for entrance animations. Exit animations are 75% of entrance duration. Use CSS transitions for simple state changes. Use `framer-motion` or the Motion library for React when choreography matters. Only animate `transform` and `opacity` (never width, height, padding, margin -- use `grid-template-rows: 0fr -> 1fr` for height). Recommended easing: `cubic-bezier(0.16, 1, 0.3, 1)` for elements entering, `cubic-bezier(0.7, 0, 0.84, 0)` for leaving. Never use bounce or elastic easing. Respect `prefers-reduced-motion` (vestibular disorders affect ~35% of adults over 40). See `references/motion-and-animation.md`.

### Text Animation
For text morphing effects (counters, status changes, tab labels), use `torph` (dependency-free, works with React/Vue/Svelte). Import as `import { TextMorph } from 'torph/react'`. Supports spring-based easing: `<TextMorph ease={{ stiffness: 200, damping: 20 }}>`.

### Sound
For UI sound feedback, use soundcn: install per sound via `npx shadcn add @soundcn/click-soft`. 700+ CC0-licensed sounds, inline base64 audio data URIs, zero runtime dependencies. Use the `useSound` hook via Web Audio API. Always provide a sound toggle. Keep sounds under 200ms. See `references/sensory-design.md`.

### Haptics
For mobile web, use `web-haptics` (`npm i web-haptics`). Built-in presets: `success` (two taps), `nudge` (strong + soft), `error` (three sharp taps), `buzz` (long vibration). Custom patterns supported. Gate behind `WebHaptics.isSupported`. Debug mode with audio for desktop testing. See `references/sensory-design.md`.

### Icons
Use Lucide React as the default icon library. For broader searches across 200K+ icons from 150+ collections, use `better-icons` CLI or MCP server. Import directly from source files (avoid barrel imports: `import Check from 'lucide-react/dist/esm/icons/check'`). Prefer outline style for UI, solid for emphasis.

### Avatars and Identity
For deterministic avatar generation from strings, use `facehash` (~3KB gzipped, SVG-based). Props: `name` (required string), `size`, `variant` ("gradient"|"solid"), `intensity3d` ("none"|"subtle"|"medium"|"dramatic"), `interactive`, `enableBlink`. Avatar/AvatarFallback compound pattern for progressive enhancement. Next.js API route handler: `toFacehashHandler()`.

### Pixel Typography
For retro/gaming/terminal aesthetics, use Geist Pixel (5 variants: Square, Grid, Circle, Triangle, Line). Install via `npm i geist`. CSS variable: `--font-geist-pixel-square`. Disable anti-aliasing: `font-smooth: never; -webkit-font-smoothing: none`.

## Step 3: Implementation Standards

### Single-File Artifacts
When building for claude.ai artifacts (HTML or React), everything goes in one file. No external CSS files, no separate JS. Inline everything. For HTML artifacts, import external scripts only from `https://cdnjs.cloudflare.com`.

### React Best Practices
Follow these rules (see `references/react-patterns.md` and `references/performance-optimization.md`):

- Server Components by default, `'use client'` only when needed
- Colocate state with the component that uses it
- Prefer composition over prop drilling
- Use `Suspense` boundaries with meaningful fallbacks
- Never store derived state, compute it during render
- Use semantic HTML elements, not `div` soup
- Eliminate waterfalls: `Promise.all()` for independent async operations
- Avoid barrel file imports (200-800ms import cost)
- Preload heavy bundles on hover/focus
- Use `content-visibility: auto` for off-screen DOM
- Minimize serialization at RSC boundaries (pass only needed fields)
- Use `React.cache()` for per-request deduplication

### CSS Variables for Theming
Always define a CSS variable system. Use semantic naming (--color-surface, --color-text, --color-accent), not literal naming (--blue, --dark-gray). Two-layer token architecture: primitives + semantic layer.

### Accessibility Defaults
Every interface must include: visible focus indicators (`:focus-visible`, 2px outline, 2px offset, 3:1 contrast), sufficient color contrast, keyboard navigability, ARIA labels on interactive elements, `alt` text on images, 24x24px minimum target size (WCAG 2.2), and dragging alternatives for drag-and-drop. Do not treat accessibility as optional. See `references/accessibility-wcag.md`.

## Step 4: Audit and Polish

Before delivering any interface, run this checklist:

1. **AI Slop Test.** Does this look like AI made it? Inter/Roboto font, purple gradients, centered everything, uniform card grids, glassmorphism, identical icon+heading+text cards? If any, fix immediately.
2. **Typography audit.** Clear hierarchy? Distinctive fonts? Body readable? Max-width on text? Vertical rhythm consistent?
3. **Color audit.** Cohesive palette? No pure black/white/gray? Accents draw the eye correctly? Light and dark modes work?
4. **Spatial audit.** Consistent spacing from scale? Proximity grouping correct (2:1+ ratio)? Squint test passes?
5. **Interaction audit.** All 8 states designed (default, hover, focus, active, disabled, loading, error, success)? Transitions smooth?
6. **Responsive audit.** Works at 375px? Scales to ultrawide? Touch targets 48px? Safe areas handled? Input method detection used?
7. **Motion audit.** Page load choreographed? Only transform/opacity animated? `prefers-reduced-motion` respected?
8. **Accessibility audit.** Tab through entire interface? Contrast sufficient? Focus trapped in modals? Skip links? ARIA patterns correct?
9. **Performance audit.** LCP < 2.5s? CLS < 0.1? INP < 200ms? Images optimized (AVIF/WebP)? Fonts preloaded?
10. **UX Writing audit.** Buttons verb-first? Errors specific and helpful? No "Submit" or "Click here"? Consistent terminology?

## Step 5: Design System Generation (DESIGN.md)

When asked to create a design system or DESIGN.md, follow the VoltAgent/Google Stitch 9-section format:

1. **Visual Theme & Atmosphere** -- Evocative prose (never "clean and modern"), mood, density, distinctive traits
2. **Color Palette & Roles** -- Every color with Semantic Name (#hex) + functional role, by category
3. **Typography Rules** -- Full hierarchy table: role, font, size, weight, line-height, letter-spacing, OpenType features
4. **Component Styling** -- Buttons, cards, inputs, navigation with all states and transitions
5. **Layout Principles** -- Spacing system, grid, container widths, whitespace philosophy, border-radius scale
6. **Depth & Elevation** -- Shadow scale table with use cases, shadow philosophy
7. **Do's and Don'ts** -- 8-10 explicit design guardrails each
8. **Responsive Behavior** -- Breakpoints, touch targets, collapsing strategy, image behavior
9. **Agent Prompt Guide** -- Quick color/font reference, example component prompts, iteration rules

See `references/design-system.md` for the complete template and 4 pre-built themes.

When building HTML presentations, every slide must fit exactly in 100vh (no scrolling). ## The Non-Negotiables

1. No design should ever look like "AI made this." No purple gradients on white. No Inter font. No centered everything. No cards nested in cards. No gray text on colored backgrounds. No glassmorphism everywhere. No identical card grids.
2. Every design must have a clear aesthetic point of view. If it could belong to any product, it belongs to none.
3. Match implementation complexity to vision. Maximalist designs need elaborate code. Minimalist designs need surgical precision.
4. Text is always a design element, never an afterthought.
5. Every detail matters. The shadow radius, the letter spacing, the hover transition duration, the border color in dark mode.
6. Every interactive element needs all 8 states: default, hover, focus, active, disabled, loading, error, success.
7. Accessibility is not optional. Focus rings, contrast ratios, keyboard navigation, screen reader support, WCAG 2.2 compliance.
8. Performance is a feature. LCP < 2.5s, CLS < 0.1, INP < 200ms.
9. UX writing is design. Buttons say what they do. Errors explain how to fix. Empty states guide the next action.
10. Vary between generations. Never converge on the same fonts, colors, or layouts across projects.
